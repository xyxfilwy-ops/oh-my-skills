# 最小经验包收集指南

经验包用于防止 Skill 只从愿望中写出来。没有经验包可以冷启动，但只能标记为 `cold-start`。

## 最小经验包

| 材料 | 数量 | 作用 | 没有时的处理 |
|---|---:|---|---|
| 成功案例 | 1-3 | 提取稳定步骤和高质量输出特征 | 标记假设，不能升到 `pro` |
| 失败案例 | 1-3 | 提取失败机制和预防规则 | 输出待观察失败点 |
| 边界案例 | 1-2 | 判断灰色地带该澄清还是部分执行 | 先写保守边界 |
| 不触发案例 | 1-2 | 防止误触发和任务抢占 | 至少根据相近任务生成 1 个反例 |
| 真实输入样本 | 1-3 | 设计输入路由和标准化结构 | 用用户描述生成模拟输入并标注假设 |
| 目标输出样本 | 1 | 定义输出契约和验收标准 | 输出模板而非稳定样式承诺 |
| 目标环境 | 1 | 决定工具、文件格式、交付路径 | 默认通用 Agent 对话 |

## 快速提问模板

每次只问 1-2 个最影响设计的问题。

### create

```text
为了把它做成可用 Skill，我先收两个关键材料：
1. 你最常给 AI 的真实输入长什么样？粘贴一个典型例子即可。
2. 它最后交回什么结果才算能直接用？
```

### review

```text
请给我当前 prompt/Skill/SOP 的正文，以及一个你担心它会误触发或输出不稳的场景。
```

### optimize

```text
请给我当前 Skill 的相关片段和 1 个真实失败案例：当时用户怎么说、Skill 怎么做错、你期望它怎么做。
```

## 经验抽取格式

```text
经验卡：
- 类型：成功 / 失败 / 边界 / 不触发
- 原始输入：
- 实际输出或期望输出：
- 关键步骤：
- 触发或不触发理由：
- 成功信号或失败信号：
- 可迁移规则：
- 不应迁移内容：
```

## 版本影响

- 无真实经验包：只能 `cold-start`。
- 至少 1 个接近真实干跑通过：可评估为 `usable`。
- 有完整最小经验包和失败诊断：可评估为 `pro`。
- 有结构、触发、干跑、selection、安装说明或 README/index 验收：可评估为 `release`。
