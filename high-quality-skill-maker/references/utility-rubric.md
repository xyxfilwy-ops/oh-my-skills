# 效用门控标准

效用门控用于判断一条规则、模块、示例、reference 或工具节点是否值得占用上下文。

## 单条规则检查

每新增或保留一条关键规则，必须回答：

| 问题 | 通过标准 | 失败信号 | 修复动作 |
|---|---|---|---|
| 它防止哪个具体失败模式？ | 能命名失败模式，如 `routing_drift` | 只能说“更专业” | 删除或改成可诊断规则 |
| 它是否告诉模型具体怎么做？ | 包含动作、条件、输出 | 只有抽象原则 | 补条件和处理方式 |
| 它是否可验证？ | 有通过标准或失败信号 | 无法判断好坏 | 改成数量、行为、证据或交付标准 |
| 它是否符合目标环境？ | 工具和文件格式真实可用 | 假设不存在的工具能力 | 改为可用工具或降级方案 |
| 它是否值得占用上下文？ | 多数执行会用到，或能防高风险错误 | 低频、装饰性、重复 | 移到 references、logs 或删除 |

## 交付级门控

```text
✅ 效用门控
- 触发是否清楚：pass / fail / partial
- 输入是否可接：pass / fail / partial
- 流程是否可执行：pass / fail / partial
- 判断是否可验证：pass / fail / partial
- 失败是否可回退：pass / fail / partial
- 输出是否可直接使用：pass / fail / partial
- 是否避免上下文膨胀：pass / fail / partial
- 是否避免过度工程：pass / fail / partial
结论：可交付 / 修改后可交付 / 不可交付
当前版本等级：cold-start / usable / pro / release
主要风险：最多 3 条
必须修复项：最多 3 条
下一轮测试建议：最多 3 条
```

## Context Budget 规则

- 每次执行必读：保留在 `SKILL.md`。
- 偶尔参考：放入 `references/`。
- 只用于维护：放入 `logs/` 或 `evals/`。
- 展示传播：放入 `index.html` 或 README。

## 删除优先级

优先删除：

1. 不绑定失败模式的规则。
2. 与其他规则重复的段落。
3. 不能改变执行结果的解释性文字。
4. 平台发布增强细节，除非用户正在做 `release`。
5. 过于理想化、和真实输入不匹配的示例。
